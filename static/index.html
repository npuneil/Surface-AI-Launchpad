You have an NPU!

Your device is a Copilot+ PC with dedicated AI silicon

Your Neural Processing Unit (NPU) is a dedicated chip designed to run AI workloads with exceptional power efficiency. Unlike running AI on your CPU (which heats up and drains battery) or GPU (which competes with display tasks), the NPU handles AI inference while barely sipping power. This means always-on AI features like Recall, Click to Do, and Semantic Search can run continuously in the background without impacting your battery life or system performance.

NPU models in Foundry Local use the QNN (Qualcomm Neural Network) execution provider, specifically optimized for the Hexagon processor in your Snapdragon chip.
💡

No NPU Detected

You can still run on-device AI using your CPU

While you don't have an NPU, Foundry Local has an excellent selection of CPU-optimized models. Your CPU handles a wider range of models (including larger ones) — the trade-off is higher power consumption and slightly slower inference compared to NPU-accelerated models. Check the Model Recommendations page for the best picks for your hardware.

Foundry Local

Microsoft's local model runtime — run SLMs entirely on your device

Run AI models locally, no cloud required

Foundry Local is an OpenAI-compatible local inference server. It manages model downloads, hardware optimization (CPU/NPU), and exposes a standard API you can use from any language.

📦

Installation

Get Foundry Local running on your machine
1

Install via winget

Open a terminal and run:

winget install Microsoft.FoundryLocal
2

Start the service

Foundry Local auto-starts when needed, but you can manually control it:

foundry service start
3

List available models

See all models optimized for your hardware (CPU + NPU if available):

foundry model list
4

Run a model interactively

Start chatting with a model directly in your terminal:

foundry model run phi-4-mini
For NPU models, add --device NPU: foundry model run qwen2.5-7b --device NPU
🔌

Using the API

OpenAI-compatible REST API for your apps

Foundry Local exposes an OpenAI-compatible API. Use it with the OpenAI Python SDK, curl, or any HTTP client. The service URL has a dynamic port — discover it with foundry service status.

Py

Python with OpenAI SDK

from openai import OpenAI

client = OpenAI(
    base_url="http://127.0.0.1:PORT/v1",
    api_key="not-needed"
)

response = client.chat.completions.create(
    model="Phi-4-mini-instruct-generic-cpu:5",
    messages=[{"role": "user", "content": "Hello!"}]
)
print(response.choices[0].message.content)
C#

C# with Microsoft.Extensions.AI

using Microsoft.Extensions.AI;

var client = new FoundryLocalClient("phi-4-mini");
var chatClient = client.CreateChatCompletionClient();

var response = await chatClient.GetResponseAsync(
    "What can you help me with?");
Console.WriteLine(response);

Recall

🔄 Find anything you've ever seen on your PC

Recall takes periodic snapshots of your screen and uses on-device AI to make everything searchable by natural language. Ask "that email about the project deadline" and Recall finds it.

🔒

Recall processes everything locally on your NPU. Screenshots never leave your device, are encrypted with Windows Hello, and stored in a protected container. You have full control over what gets saved and can delete snapshots at any time.

🧠

How It Works Under the Hood

The NPU does the heavy lifting
📸

Screen Capture

Periodic screenshots are taken (every ~5 seconds when content changes). Images are immediately encrypted before any processing.

📝

NPU-Powered OCR

The NPU runs optical character recognition on each snapshot, extracting all visible text with minimal power draw. This enables text search across screenshots.

🔗

Semantic Embedding

Extracted text is converted into semantic vectors (embeddings) by the NPU, enabling meaning-based search — so "vacation photos" finds screenshots of beach pictures even if those words never appeared on screen.

Semantic Search

Search by meaning, not just keywords — powered by on-device AI

🔎 "Find that document about budget planning"

Windows Semantic Search understands what you mean, not just what you type. It uses the NPU to generate embeddings for your files, so you can search by concept instead of exact filenames.

🧠

How Semantic Search Works

AI-enhanced Windows Search
1

Content Indexing

Windows scans your documents, emails, and files. The NPU generates semantic embeddings — numerical vectors that capture meaning — for each piece of content. This happens in the background with negligible performance impact thanks to NPU efficiency.

2

Natural Language Queries

When you search, your query is also converted to an embedding. The system finds documents whose meaning is closest to your query — even if they don't contain the exact words you typed.

3

Ranked Results

Results are ranked by semantic similarity, combining traditional keyword matching with AI understanding. This means a search for "vacation plans" can find a file named "summer-trip-itinerary.docx" even though those words don't overlap.

Edge AI Architecture

How on-device AI systems are structured

In a traditional architecture, data flows from edge devices to cloud servers and back. In Edge AI, the entire inference pipeline runs locally: 

Traditional:  Sensors → Network → Cloud GPU → Network → Response
Edge AI:      Sensors → Local NPU/CPU → Immediate Response

Key Enabling Technologies:
├── Model Quantization (FP32 → INT4/INT8, 4-8x size reduction)
├── Model Pruning (remove unnecessary connections, 50%+ compression)
├── Knowledge Distillation (train small models from large ones)
├── NPU Hardware (45+ TOPS at <5W power draw)
└── Optimized Runtimes (ONNX Runtime, DirectML, QNN)
The key insight: Edge AI doesn't replace cloud AI — it complements it. Use edge for real-time, privacy-sensitive, and latency-critical tasks. Use cloud for training, complex reasoning, and tasks requiring the largest models.

Model Optimization

Making AI models smaller, faster, and more power-efficient for edge deployment

From data center to device: the art of model optimization

Model optimization is the bridge between powerful AI capabilities and the constraints of edge hardware. Through quantization, pruning, and distillation, models can be compressed 4-8x while retaining 90-99% of their original quality.

📐

Quantization

Reducing numerical precision for massive efficiency gains

Quantization reduces the precision of model weights from 32-bit floats to smaller representations. This directly reduces memory usage and speeds up inference — especially on NPUs designed for integer math.

PrecisionBitsMemory SavingsQuality ImpactBest For
FP32 (Full)32BaselineNoneTraining, reference
FP16 (Half)162x reductionMinimalGPU inference
INT884x reduction~1-2% lossCPU/NPU inference
INT448x reduction~2-5% lossNPU-optimized models
1.58-bit (BitNET)~1.620x reduction~5-10% lossUltra-efficient edge
NPU models in Foundry Local (like qwen2.5-7b-instruct-qnn-npu) use INT4 quantization optimized for the Qualcomm Hexagon processor. This is why a 7B model fits in just 2.78 GB.
🔧

Quantization Techniques

Different approaches for different needs
PTQ

Post-Training Quantization

Applied after training without re-training. Fastest to implement — just convert an existing model. Slight quality loss. Most Foundry Local models use this approach.

QAT

Quantization-Aware Training

Simulates quantization during training so the model adapts to lower precision. Better quality than PTQ but requires re-training. Used for production-critical models.

DYN

Dynamic Quantization

Weights are quantized statically, but activations are quantized at runtime based on actual data ranges. Good balance of quality and efficiency without calibration data.

Microsoft Olive

Hardware-aware model optimization toolkit

Microsoft Olive is an optimization toolkit with 40+ built-in optimization passes that automatically selects the best optimizations for your target hardware — whether that's an NPU, GPU, or CPU. It supports Phi, Qwen, Llama, Gemma, and more out of the box. 

# Install Microsoft Olive
pip install olive-ai

# Optimize a model for your hardware
olive auto-opt \
    --model_name_or_path microsoft/Phi-4-mini-instruct \
    --device cpu \
    --provider CPUExecutionProvider \
    --output_path optimized_model/

# For NPU targets (Qualcomm QNN)
olive auto-opt \
    --model_name_or_path microsoft/Phi-4-mini-instruct \
    --device npu \
    --provider QNNExecutionProvider \
    --output_path optimized_model_npu/
Olive can achieve 2-6x speedups with 60-75% size reduction through automatic optimization. It's the recommended tool for preparing models for Foundry Local.

DirectML & NPU Acceleration

How Windows routes AI to the right hardware

DirectML is the hardware abstraction layer that makes cross-silicon development possible. When you load a model through Windows ML or ONNX Runtime, DirectML automatically selects the optimal execution provider. 

// C# — Using ONNX Runtime with DirectML for NPU
using Microsoft.ML.OnnxRuntime;

var sessionOptions = new SessionOptions();
// DirectML automatically picks NPU if available
sessionOptions.AppendExecutionProvider_DmlExecutionProvider();

using var session = new InferenceSession("model.onnx", sessionOptions);

// Create input tensor
var inputTensor = new DenseTensor<float>(new[] { 1, 3, 224, 224 });
var inputs = new List<NamedOnnxValue> 
{
    NamedOnnxValue.CreateFromTensor("input", inputTensor) 
};

// Run inference — hardware selected automatically
using var results = session.Run(inputs);
You don't need to detect hardware or write platform-specific code. DirectML handles hardware selection automatically. Your code runs the same on Snapdragon ARM64 and Intel x64 machines.

AI Agents & Tools

Build autonomous AI that reasons, plans, and takes action — all running locally

🤖 From chatbots to autonomous agents

AI agents go beyond simple Q&A. They can reason about tasks, plan multi-step workflows, call external functions, and coordinate with other agents — all powered by local SLMs running on your device via Foundry Local.

🧠

What Makes an AI Agent?

Beyond simple prompt → response
1

Reasoning

The agent analyzes the task and breaks it into steps. Models like DeepSeek R1 and Phi-4-mini-reasoning excel at chain-of-thought planning.

2

Tool Use (Function Calling)

The model decides which tools to invoke — search the web, query a database, read a file, call an API. Foundry Local models with tools support can generate structured function calls.

3

Observation & Iteration

After calling a tool, the agent observes the result and decides the next action. This reason → act → observe loop continues until the task is complete.

4

Memory & Context

Agents maintain conversation history and can reference earlier steps. Local execution means this context never leaves your device.

🔧

Function Calling with Foundry Local

Give your local model the ability to call functions

Models with tools support (like Qwen2.5-7B and Phi-4-mini) can generate structured function calls. Use the OpenAI-compatible API: 

from openai import OpenAI

client = OpenAI(base_url="http://127.0.0.1:PORT/v1", api_key="not-needed")

# Define tools the model can call
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a location",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City name"}
            },
            "required": ["location"]
        }
    }
}]

response = client.chat.completions.create(
    model="qwen2.5-7b-instruct-qnn-npu:2",
    messages=[{"role": "user", "content": "What's the weather in Seattle?"}],
    tools=tools,
    tool_choice="auto"
)

# Model returns a tool call instead of text
tool_call = response.choices[0].message.tool_calls[0]
print(f"Call: {tool_call.function.name}({tool_call.function.arguments})")
Function calling works the same whether you're using an NPU model or a CPU model. The key is using a model alias that lists "tools" in its supported tasks.

Knowledge Bases with Foundry IQ

Give agents access to your documents with agentic retrieval

Foundry IQ is the managed knowledge layer that connects your data (Azure Storage, SharePoint, OneLake, web) to agents — with automatic chunking, embeddings, indexing, and agentic retrieval. 

Architecture:
Documents (PDF, Word, MD) → Azure Storage
    → Knowledge Source (auto-chunk + embed)
    → Azure AI Search (vector index)
    → Knowledge Base (agentic retrieval)
    → Agent (RAG with citations)
ConceptWhat It Does
Knowledge SourceConnection to your data store. Foundry IQ auto-chunks documents, generates embeddings, and indexes in AI Search.
Knowledge BaseOrchestrates agentic retrieval — breaks complex queries into sub-queries, searches in parallel, reranks, returns citations.
Agentic RetrievalMulti-query pipeline with query decomposition, parallel execution, semantic reranking, and unified answers with source citations.
A knowledge base can contain multiple knowledge sources (Blob Storage + SharePoint + web), and multiple agents can share the same knowledge base. Agentic retrieval respects permissions (ACLs) and Microsoft Purview sensitivity labels.

Foundry Evaluators for Edge Models

Validate SLM quality on Copilot+ PCs using cloud evaluation

The key challenge with edge deployment: how do you know your local SLM is good enough? Foundry evaluators solve this by letting you systematically test edge model outputs against cloud baselines.

1

Generate Edge Outputs

Run your test dataset through the local SLM on a Copilot+ PC using Foundry Local. Capture outputs via the OpenAI-compatible API or batch inference through AI Toolkit.

foundry model run phi-4-mini --device CPU < test_prompts.jsonl > edge_outputs.jsonl
2

Evaluate with Foundry

Upload edge outputs to Foundry and run evaluators — groundedness, relevance, coherence, fluency, and custom metrics. Compare against GPT-4o baseline outputs.

from azure.ai.evaluation import EvaluatorConfig, evaluate

# Evaluate edge model outputs against cloud baseline
results = evaluate(
    data="edge_outputs.jsonl",
    evaluators={
        "groundedness": EvaluatorConfig(
            column_mapping={"query": "prompt", "response": "output"}
        ),
        "relevance": EvaluatorConfig(
            column_mapping={"query": "prompt", "response": "output"}
        ),
        "coherence": EvaluatorConfig(
            column_mapping={"response": "output"}
        ),
    },
    azure_ai_project={
        "subscription_id": "<sub-id>",
        "resource_group_name": "<rg>",
        "project_name": "<project>"
    }
)

# Compare scores: edge vs cloud
print(f"Edge groundedness: {results['groundedness']:.2f}")
print(f"Edge relevance:    {results['relevance']:.2f}")
print(f"Edge coherence:    {results['coherence']:.2f}")
3

Set Quality Gates

Define minimum thresholds — e.g., edge model must score ≥90% of GPT-4o on groundedness and ≥85% on relevance. Automate this in CI/CD to catch quality regressions when you update models.

4

Iterate and Optimize

If the edge model falls short, try a larger SLM (7B → 14B), apply fine-tuning with LoRA via AI Toolkit, or use knowledge distillation from GPT-4o. Re-evaluate until quality gates pass.

Foundry evaluators run in the cloud but evaluate any model output — including outputs from local Foundry Local models on Copilot+ PCs. This lets you maintain cloud-grade quality standards for edge deployments.
🔄

Cloud-to-Edge Deployment Pipeline

From cloud model to NPU-optimized SLM
☁️

1. Develop in the Cloud

Build your AI feature using GPT-4o in Foundry. Perfect prompts, add knowledge bases, test with evaluators. The cloud version is your quality baseline.

🧪

2. Distill to SLM

Use GPT-4o outputs as training data for a smaller model. Fine-tune Phi-4-mini or Qwen2.5-7B with LoRA using AI Toolkit. The student model learns the teacher's behavior for your specific domain.

⚙️

3. Optimize for NPU

Run Microsoft Olive for hardware-aware quantization (INT4 for NPU, INT8 for CPU). Compile through QNN for Snapdragon devices. Validate with Foundry evaluators at each step.

olive auto-opt --model_name_or_path ./fine-tuned-model --device npu --provider QNNExecutionProvider --output_path ./npu-model/
💻

4. Deploy to Copilot+ PCs

Distribute the optimized model via Foundry Local or bundle it with your application. Users get sub-10ms local inference, full offline capability, and complete data privacy — with cloud-validated quality.
