What Are CPU, GPU, and NPU?

Understanding the three AI-capable processors in your PC
⚙️

CPU — Central Processing Unit

The general-purpose brain of your PC. CPUs handle everything from running your OS to executing application logic. For AI, CPUs offer the widest model compatibility — any model that runs anywhere will run on a CPU. The trade-off: higher power consumption and slower inference compared to specialized hardware. Foundry Local's CPU models use optimized ONNX Runtime with AVX/AVX-512 instructions.

🖥️

GPU — Graphics Processing Unit

Originally designed for rendering graphics, GPUs excel at massively parallel computation — making them powerful for AI training and inference on large models. GPUs deliver the highest raw throughput but consume significant power and compete with display rendering tasks. DirectML enables GPU-accelerated inference on Windows across vendors (Intel, NVIDIA, AMD).

🧠

NPU — Neural Processing Unit

A dedicated AI accelerator chip purpose-built for neural network inference. NPUs deliver 40+ TOPS (Trillion Operations Per Second) at a fraction of the power draw of a CPU or GPU — typically under 5 watts. This enables always-on AI features that run continuously without draining your battery. NPUs use fixed-function hardware and specialized data paths optimized for the matrix multiplications and convolutions that dominate AI workloads.

Foundry Local

Microsoft's local model runtime — run SLMs entirely on your device

Purpose-built for shipping AI applications

Foundry Local is more than a local inference server — it's an SDK for shipping AI-powered applications. It works fully offline with no cloud dependencies, ships SDKs for Python, JavaScript, C#, and Rust, exposes an OpenAI-compatible API for drop-in integration, and is hardware-optimized in partnership with silicon vendors for NPU, GPU, and CPU execution. Everything stays on-device.

📦

Get Foundry Local running on your machine
1

Install via winget

Open a terminal (press Win + X → Terminal, or search for PowerShell or Command Prompt) and run:

winget install Microsoft.FoundryLocal
2

Start the service

Foundry Local auto-starts when needed, but you can manually control it:

foundry service start
3

List available models

See all models optimized for your hardware (CPU + NPU if available):

foundry model list
4

Run a model interactively

Start chatting with a model directly in your terminal:

foundry model run phi-4-mini

Using the API

OpenAI-compatible REST API for your apps

Foundry Local exposes an OpenAI-compatible API. Use it with the OpenAI Python SDK, curl, or any HTTP client. The service URL has a dynamic port — discover it by pasting the following command in your terminal:

foundry service status
Py

from openai import OpenAI

client = OpenAI(
    base_url="http://127.0.0.1:PORT/v1",
    api_key="not-needed"
)

    model="Phi-4-mini-instruct-generic-cpu:5",
    messages=[{"role": "user", "content": "Hello!"}]
)
print(response.choices[0].message.content)

var chatClient = client.CreateChatCompletionClient();

    "What can you help me with?");

How It Works Under the Hood

The NPU does the heavy lifting
📸

Screen Capture

Periodic screenshots are taken (every ~5 seconds when content changes). Images are immediately encrypted before any processing.

📝

NPU-Powered OCR

The NPU runs optical character recognition on each snapshot, extracting all visible text with minimal power draw. This enables text search across screenshots.

🔗

Semantic Embedding

Extracted text is converted into semantic vectors (embeddings) by the NPU, enabling meaning-based search — so "vacation photos" finds screenshots of beach pictures even if those words never appeared on screen.

Windows Semantic Search understands what you mean, not just what you type. It uses the NPU to generate embeddings for your files, so you can search by concept instead of exact filenames.

🧠

How Semantic Search Works

AI-enhanced Windows Search
1

Content Indexing

Windows scans your documents, emails, and files. The NPU generates semantic embeddings — numerical vectors that capture meaning — for each piece of content. This happens in the background with negligible performance impact thanks to NPU efficiency.

2

Natural Language Queries

When you search, your query is also converted to an embedding. The system finds documents whose meaning is closest to your query — even if they don't contain the exact words you typed.

3

Ranked Results

Results are ranked by semantic similarity, combining traditional keyword matching with AI understanding. This means a search for "vacation plans" can find a file named "summer-trip-itinerary.docx" even though those words don't overlap.

Edge AI Architecture

How on-device AI systems are structured

In a traditional architecture, data flows from edge devices to cloud servers and back. In Edge AI, the entire inference pipeline runs locally: 

Traditional:  Sensors → Network → Cloud GPU → Network → Response
Edge AI:      Sensors → Local NPU/CPU → Immediate Response

Key Enabling Technologies:
├── Model Quantization (FP32 → INT4/INT8, 4-8x size reduction)
├── Model Pruning (remove unnecessary connections, 50%+ compression)
├── Knowledge Distillation (train small models from large ones)
├── NPU Hardware (45+ TOPS at <5W power draw)
└── Optimized Runtimes (ONNX Runtime, DirectML, QNN)
The key insight: Edge AI doesn't replace cloud AI — it complements it. Use edge for real-time, privacy-sensitive, and latency-critical tasks. Use cloud for training, complex reasoning, and tasks requiring the largest models.

Model Optimization

Making AI models smaller, faster, and more power-efficient for edge deployment

From data center to device: the art of model optimization

Model optimization is the bridge between powerful AI capabilities and the constraints of edge hardware. Through quantization, pruning, and distillation, models can be compressed 4-8x while retaining 90-99% of their original quality.

📐

Quantization

Reducing numerical precision for massive efficiency gains

Quantization reduces the precision of model weights from 32-bit floats to smaller representations. This directly reduces memory usage and speeds up inference — especially on NPUs designed for integer math.

PrecisionBitsMemory SavingsQuality ImpactBest For
FP32 (Full)32BaselineNoneTraining, reference
FP16 (Half)162x reductionMinimalGPU inference
INT884x reduction~1-2% lossCPU/NPU inference
INT448x reduction~2-5% lossNPU-optimized models
1.58-bit (BitNET)~1.620x reduction~5-10% lossUltra-efficient edge
NPU models in Foundry Local (like qwen2.5-7b-instruct-qnn-npu) use INT4 quantization optimized for the Qualcomm Hexagon processor. This is why a 7B model fits in just 2.78 GB.
🔧

Quantization Techniques

Different approaches for different needs
PTQ

Post-Training Quantization

Applied after training without re-training. Fastest to implement — just convert an existing model. Slight quality loss. Most Foundry Local models use this approach.

QAT

Quantization-Aware Training

Simulates quantization during training so the model adapts to lower precision. Better quality than PTQ but requires re-training. Used for production-critical models.

DYN

Dynamic Quantization

Weights are quantized statically, but activations are quantized at runtime based on actual data ranges. Good balance of quality and efficiency without calibration data.

Microsoft Olive

Hardware-aware model optimization toolkit

Microsoft Olive is an optimization toolkit with 40+ built-in optimization passes that automatically selects the best optimizations for your target hardware — whether that's an NPU, GPU, or CPU. It supports Phi, Qwen, Llama, Gemma, and more out of the box.

Olive can achieve 2-6x speedups with 60-75% size reduction through automatic optimization. It's the recommended tool for preparing models for Foundry Local.

Function Calling with Foundry Local

Give your local model the ability to call functions

Models with tools support (like Qwen2.5-7B and Phi-4-mini) can generate structured function calls. Use the OpenAI-compatible API: 

from openai import OpenAI

    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a location",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City name"}
            },
            "required": ["location"]
        }
    }
}]

response = client.chat.completions.create(
    model="qwen2.5-7b-instruct-qnn-npu:2",
    messages=[{"role": "user", "content": "What's the weather in Seattle?"}],
    tools=tools,
    tool_choice="auto"
)

# Model returns a tool call instead of text
tool_call = response.choices[0].message.tool_calls[0]

Foundry Evaluators for Edge Models

Validate SLM quality on Copilot+ PCs using cloud evaluation

The key challenge with edge deployment: how do you know your local SLM is good enough? Foundry evaluators solve this by letting you systematically test edge model outputs against cloud baselines.

1

Generate Edge Outputs

Run your test dataset through the local SLM on a Copilot+ PC using Foundry Local. Capture outputs via the OpenAI-compatible API or batch inference through AI Toolkit.

foundry model run phi-4-mini --device CPU < test_prompts.jsonl > edge_outputs.jsonl
2

Evaluate with Foundry

Upload edge outputs to Foundry and run evaluators — groundedness, relevance, coherence, fluency, and custom metrics. Compare against GPT-4o baseline outputs.

from azure.ai.evaluation import EvaluatorConfig, evaluate

# Evaluate edge model outputs against cloud baseline
results = evaluate(
    data="edge_outputs.jsonl",
    evaluators={
        "groundedness": EvaluatorConfig(
            column_mapping={"query": "prompt", "response": "output"}
        ),
        "relevance": EvaluatorConfig(
            column_mapping={"query": "prompt", "response": "output"}
        ),
        "coherence": EvaluatorConfig(
            column_mapping={"response": "output"}
        ),
    },
    azure_ai_project={
        "subscription_id": "<sub-id>",
        "resource_group_name": "<rg>",
        "project_name": "<project>"
    }
)

# Compare scores: edge vs cloud
print(f"Edge groundedness: {results['groundedness']:.2f}")
print(f"Edge relevance:    {results['relevance']:.2f}")
print(f"Edge coherence:    {results['coherence']:.2f}")

Define minimum thresholds — e.g., edge model must score ≥90% of GPT-4o on groundedness and ≥85% on relevance. Automate this in CI/CD to catch quality regressions when you update models.

4

If the edge model falls short, try a larger SLM (7B → 14B), apply fine-tuning with LoRA via AI Toolkit, or use knowledge distillation from GPT-4o. Re-evaluate until quality gates pass.

Foundry evaluators run in the cloud but evaluate any model output — including outputs from local Foundry Local models on Copilot+ PCs. This lets you maintain cloud-grade quality standards for edge deployments.

What Makes a Great Bolt-On NPU App?

Patterns ISVs follow for NPU integration
⚡

Always-On AI

The NPU's low power draw means AI features can run continuously without impacting battery life. ISVs design features that are "always analyzing" — background noise suppression, real-time translation, or continuous document indexing.

🔒

Privacy by Default

Processing on the NPU means sensitive data never leaves the device. ISVs use this to offer AI features in regulated industries (healthcare, finance, legal) where cloud processing isn't allowed.

🎯

Offload from GPU/CPU

By routing AI workloads to the NPU, apps free up the GPU for rendering and the CPU for application logic. This is why video editors can run AI effects while encoding — the workloads don't compete.
